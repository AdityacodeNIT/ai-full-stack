2. AUTHENTICATION SYSTEM (CLERK)
================================================================================

IMPLEMENTATION:
- Clerk handles all authentication (signup, login, session management)
- JWT tokens for API authentication
- Clerk userId (string like "user_xxx") used throughout system
- No password storage in MongoDB

USER FLOW:
1. User signs up via Clerk
2. Webhook syncs user to MongoDB (optional)
3. First user automatically becomes admin
4. Subsequent users are regular users
5. Admin can change roles via dashboard

ROLE MANAGEMENT:
- Roles stored in Clerk publicMetadata.role (source of truth)
- Roles also stored in MongoDB for analytics
- Two roles: "admin" and "user"
- Admin has access to admin dashboard

ENDPOINTS:
- GET /api/auth/me - Get current user info, auto-assign role on first login
- GET /api/auth/ws-token - Get Clerk JWT for WebSocket authentication
- POST /api/webhooks/clerk - Sync Clerk users to MongoDB

MIDDLEWARE:
- requireAuth() - Ensures user is authenticated
- attachClerkUserId - Attaches req.clerkUserId from Clerk token
- isAdmin() - Checks if user has admin role

3. INTERVIEW SYSTEM
================================================================================

INTERVIEW CREATION:
1. User fills form: role, level, tech stack, question type, number of questions
2. Frontend creates interview via POST /api/interviews
3. Interview saved to MongoDB with status "created"
4. User redirected to interview session page

INTERVIEW SESSION:
1. WebSocket connection established with Clerk JWT
2. Backend validates JWT and extracts clerkUserId
3. Interview agent initialized with Gemini AI
4. Questions generated based on interview context
5. User answers via voice (AssemblyAI transcription)
6. AI analyzes each response in real-time
7. Final report generated at end

INTERVIEW FLOW:
- User starts interview → WebSocket connects
- AI asks question → TTS speaks question
- User answers → AssemblyAI transcribes
- User submits → AI analyzes response
- Next question → Repeat
- All questions done → AI generates final report
- Interview marked as "completed"

AI AGENT:
- Uses Google Gemini 1.5 Flash model
- Generates questions based on role, level, tech stack
- Analyzes responses for confidence, clarity, technical understanding
- Provides real-time feedback after each question
- Generates comprehensive final report

EXPERIENCE LEVELS:
- Junior: Basic concepts, simple scenarios
- Mid-level: Intermediate concepts, practical experience
- Senior: Advanced concepts, system design, leadership
- Lead/Executive: Strategic thinking, team management

QUESTION TYPES:
- Technical: Coding, algorithms, system design
- Behavioral: Past experiences, soft skills
- Balanced: Mix of technical and behavioral

4. PROCTORING SYSTEM
================================================================================

OVERVIEW:
- Client-side ML detection (no video upload)
- Silent operation (user not notified)
- Violations saved to backend with confidence scores
- Admin can review violations in dashboard

DETECTORS:
1. Face Detection (MediaPipe)
   - Detects: no face, multiple faces
   - Runs every 5 seconds
   - 10 second cooldown between violations

2. Object Detection (COCO-SSD)
   - Detects: cell phone, book, laptop, etc.
   - Only sends violations with ≥60% confidence
   - Runs every 5 seconds
   - 10 second cooldown between violations

ARCHITECTURE:
- BaseProctoringDetector: Abstract base class
- FaceDetector: Extends base, implements face detection
- ObjectDetector: Extends base, implements object detection
- ProctoringManager: Orchestrates multiple detectors
- ModelCache: Singleton cache for TensorFlow models

MODEL CACHING:
- Models loaded once and cached in memory
- First interview: 8-10 seconds to load models
- Subsequent interviews: Instant (cached)
- Models shared across all interviews
- Prevents remounting and reloading

VIOLATION FLOW:
1. Detector detects violation (e.g., phone with 82% confidence)
2. If confidence ≥60%, send to backend via WebSocket
3. Backend saves violation with metadata to MongoDB
4. User continues interview (not notified)
5. Admin reviews violations in dashboard

VIOLATION DATA:
{
  type: "prohibited_object_cell_phone",
  detector: "ObjectDetector",
  timestamp: 1234567890,
  questionNumber: 3,
  metadata: {
    object: "cell phone",
    confidence: 0.82,
    bbox: [x, y, width, height]
  },
  silent: true
}

CONFIGURATION:
- confidenceThreshold: 0.6 (60% minimum)
- prohibitedObjects: ['cell phone', 'book']
- checkInterval: 5000 (check every 5 seconds)
- cooldownPeriod: 10000 (10 second cooldown)

9. WEBSOCKET PROTOCOL
================================================================================

CONNECTION:
- URL: ws://localhost:3000?token=<clerk_jwt>
- Backend decodes JWT and extracts clerkUserId
- Connection authenticated and stored

MESSAGE TYPES:

1. start (Client → Server)
   { type: "start", interviewId: "..." }
   - Starts interview session
   - Initializes AI agent

2. response (Client → Server)
   { type: "response", response: "user's answer" }
   - Submits user's answer
   - Triggers AI analysis

3. question (Server → Client)
   { type: "question", question: "...", questionNumber: 1, totalQuestions: 10 }
   - AI sends next question

4. analysis (Server → Client)
   { type: "analysis", analysis: { confidence: "High", clarity: "High", ... } }
   - AI sends feedback on answer

5. finalReport (Server → Client)
   { type: "finalReport", report: { overallScore: 85, ... } }
   - AI sends final report

6. proctoring_violation (Client → Server)
   { 
     type: "proctoring_violation",
     violationType: "prohibited_object_cell_phone",
     detectorName: "ObjectDetector",
     timestamp: 1234567890,
     metadata: { object: "cell phone", confidence: 0.82 },
     silent: true
   }
   - Proctoring violation detected

7. end (Server → Client)
   { type: "end" }
   - Interview completed

8. error (Server → Client)
   { type: "error", message: "..." }
   - Error occurred

10. DEPLOYMENT GUIDE
================================================================================

ENVIRONMENT VARIABLES:

Backend (.env):
- MONGODB_URI=mongodb://...
- CLERK_SECRET_KEY=sk_...
- CLERK_PUBLISHABLE_KEY=pk_...
- GEMINI_API_KEY=...
- PORT=3000

Frontend (.env):
- VITE_CLERK_PUBLISHABLE_KEY=pk_...
- VITE_API_URL=http://localhost:3000
- VITE_WS_URL=ws://localhost:3000

BUILD COMMANDS:
Frontend: npm run build (in ai-ticketing/)
Backend: No build needed (Node.js)

DEPLOYMENT:
1. Deploy backend to any Node.js hosting (Heroku, Railway, etc.)
2. Deploy frontend to Vercel, Netlify, etc.
3. Update environment variables
4. Configure Clerk webhook URL
5. Test WebSocket connection

CLERK SETUP:
1. Create Clerk application
2. Get publishable and secret keys
3. Configure webhook endpoint: POST /api/webhooks/clerk
4. Add webhook signing secret to .env
5. Enable user.created, user.updated, user.deleted events

MONGODB SETUP:
1. Create MongoDB Atlas cluster
2. Get connection string
3. Add to MONGODB_URI in .env
4. Whitelist deployment IP

GEMINI API SETUP:
1. Get API key from Google AI Studio
2. Add to GEMINI_API_KEY in .env
